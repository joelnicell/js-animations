Animation techniques for the web

The tick boxes for effective websites with animations are:

    • Highly semantic
      Fast

    • We can rate each of these approaches to see how well they match up in these areas. As well as seeing the animating power of each approach.

    Example 1: Javascript updates css

    In this example, a box div is having it's css style attribute updated by an animation loop from Javascript.

    Here is the code snippet:

    				
						const box = document.getElementById("box");
		
						let value = 0;
						function animate(t) {
							value = Math.sin(t * 0.001) * 40 + 100;
							box.style.width = `${value}px`;
							requestAnimationFrame((t) => animate(t))
						}
		
						animate();

    What are the pros to this approach?

    • Updates are controlled by the well understood and easy-to-understand animation cycle
      State is controlled inside of animation cycle.
      One line of code is directly impacting html css, so not only is the processing 'native', but also keeps semantic relationships in tact
      Easy to manage animation cycle, and is relatively easy to inject interactivity with event listeners etc. in vanilla JS
      Not re-writing HTML, only changing styles via css

    • What are the cons to this approach?

    • Value's constantly changing in the DOM tree. Opening the command palette sees updates occuring regularly
      limited by the styling of css. Stays within the 'box' model.
      The threading inside of a single animation loop is singular, even though the loop itself is asynchronous. This means animations could slow down

    • Keeping semantics in tact:

    This example seems to have a pretty tight relationship between a user's interaction with the page, and the state of the animation.

    Final review:

    This approach seems ideal for interactive animated elements. Not only because it ticks semantic boxes in it's purest sense, but also because in general, interactive elements (buttons, scrolling events etc.) should not be over-complicated by animations anyway. The button above is already really annoying for users to try and click. Why would you want MORE animation on an interactive element?

    I think if this technique can be pushed to it's limits, then it could prove highly effective.
